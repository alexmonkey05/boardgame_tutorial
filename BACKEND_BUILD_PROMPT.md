# 보드게임 카페 앱 백엔드 구축 프롬프트

당신은 모바일 앱 백엔드 아키텍트이자 API 설계자입니다. 아래 목표와 현재 프로토타입의 기능을 바탕으로, 보드게임 카페 도우미 앱의 백엔드를 설계하고 구현하기 위한 구체적인 기술 기획안과 개발 작업 목록을 작성해주세요.

## 서비스 목표

이 서비스는 보드게임 카페에 방문한 사용자가 현장에서 빠르게 보드게임을 찾고, 이해하고, 선택하도록 돕는 모바일 앱의 백엔드입니다.

앱의 핵심 사용 상황은 다음 세 가지입니다.

1. 사용자가 보드게임 박스나 구성품을 촬영하면 어떤 게임인지 식별합니다.
2. 사용자의 이전 검색, 조회, 플레이 기록을 기반으로 현재 카페에서 바로 할 수 있는 게임을 추천합니다.
3. 사용자가 현재 방문 중인 보드게임 카페의 보유 게임 목록을 검색, 필터링, 정렬할 수 있게 합니다.

백엔드는 앱이 직접 많은 정보를 들고 있지 않도록, 보드게임 설명 데이터, 카페별 보유 목록, 사용자 행동 기록, 이미지 인식 결과, 추천에 필요한 데이터를 안정적으로 제공해야 합니다.

## 현재 프론트엔드 프로토타입 상태

현재 앱은 단일 `index.html` 프로토타입으로 구현되어 있습니다.

- 카메라 인식은 실제 카메라/API 연동이 아니라 버튼 기반 시뮬레이션입니다.
- 게임, 카페, 카페별 보유 목록, 사용자 기록은 프론트엔드 코드 안에 하드코딩되어 있습니다.
- 추천 로직은 클라이언트에서 점수화 방식으로 동작합니다.
- 카페 인식은 QR/근처 매장 버튼이 있지만 실제 QR 스캔이나 위치 API는 없습니다.

백엔드 구축의 목표는 이 하드코딩된 데이터를 서버 데이터로 분리하고, 실제 앱 또는 웹 프론트엔드가 API를 통해 사용할 수 있는 구조를 만드는 것입니다.

## 백엔드가 해결해야 할 핵심 문제

### 1. 보드게임 마스터 데이터 관리

보드게임별 기본 정보를 서버에 저장하고 제공해야 합니다.

필수 데이터 예시는 다음과 같습니다.

- 게임 ID
- 한국어 이름
- 영문 이름 또는 별칭
- 간략 설명
- 규칙 요약
- 추천 인원 최소/최대
- 평균 플레이 시간
- 난이도
- 장르
- 태그
- 초보자 추천 여부
- 아이와 함께 가능 여부
- 파티 게임 여부
- 전략 게임 여부
- 협력/경쟁 성향
- 이미지 URL
- 확장판 또는 비슷한 이름의 게임 관계

게임 상세 설명은 너무 길지 않게, 보드게임 카페 현장에서 빠르게 판단할 수 있는 형태로 제공되어야 합니다.

### 2. 카페 및 카페별 보유 게임 관리

각 보드게임 카페의 기본 정보와 보유 게임 목록을 관리해야 합니다.

필수 데이터 예시는 다음과 같습니다.

- 카페 ID
- 카페 이름
- 지점명
- 주소
- 위치 좌표
- QR 코드 또는 매장 식별자
- 영업 상태
- 카페별 보유 게임 목록
- 게임별 선반 위치
- 대여 가능 여부 또는 임시 품절 여부
- 카페별 추천/인기 게임 메타데이터

앱은 현재 사용자가 있는 카페를 기준으로 게임 목록과 추천 결과를 보여줘야 하므로, 카페 식별과 보유 목록 API가 중요합니다.

### 3. 이미지 기반 보드게임 식별

사용자가 촬영한 보드게임 사진을 받아 어떤 보드게임인지 판별해야 합니다.

백엔드가 직접 AI 모델을 운영하거나 외부 이미지 인식 API를 호출할 수 있습니다. 어떤 방식을 선택하든 API 응답은 앱에서 사용하기 쉬운 형태여야 합니다.

응답에는 다음이 포함되어야 합니다.

- 가장 가능성 높은 게임 ID
- 신뢰도
- 후보 게임 목록
- 각 후보의 신뢰도
- 현재 카페 보유 여부
- 다시 촬영이 필요한지 여부
- 사용자에게 보여줄 짧은 안내 문구

신뢰도가 낮거나 후보가 여러 개인 경우, 단일 정답을 강요하지 말고 후보 선택 흐름으로 연결해야 합니다.

### 4. 사용자 행동 기록 저장

추천 품질을 높이기 위해 사용자의 앱 사용 기록을 저장해야 합니다.

기록 대상 예시는 다음과 같습니다.

- 게임 검색
- 게임 상세 조회
- 카메라 인식 결과 확인
- 후보 게임 중 사용자가 선택한 게임
- 추천 결과 클릭
- 플레이 완료로 표시한 게임
- 싫어요/관심 없음 처리한 게임
- 현재 방문 카페
- 인원수, 남은 시간, 원하는 분위기 같은 추천 조건

로그는 추천에 사용할 수 있어야 하지만, 개인정보를 과도하게 수집하지 않도록 익명 사용자 또는 선택적 로그인 구조도 고려해주세요.

### 5. 추천 API

추천 API는 사용자의 이전 기록과 현재 상황, 현재 카페의 보유 게임 목록을 함께 고려해야 합니다.

추천 입력 예시는 다음과 같습니다.

- 사용자 ID 또는 익명 세션 ID
- 현재 카페 ID
- 인원수
- 남은 시간
- 원하는 분위기
- 선호 난이도
- 제외할 게임
- 이전에 플레이한 게임

추천 응답에는 다음이 포함되어야 합니다.

- 추천 게임 목록
- 추천 점수 또는 우선순위
- 추천 이유
- 현재 카페 보유 여부
- 선반 위치
- 조건에 맞지 않지만 추천할 만한 대체 후보

추천 이유는 사용자가 이해하기 쉬운 문장이어야 합니다. 예를 들어 "4명이 하기 좋아요", "30분 안에 끝나요", "이전에 본 전략 게임과 비슷해요", "지금 이 카페에 있어요"처럼 표현해주세요.

## 필요한 API 초안

아래 API 목록을 기준으로 REST 또는 GraphQL 중 적절한 방식을 선택해 설계해주세요. 특별한 이유가 없다면 모바일 앱 MVP에는 REST API를 우선 고려해주세요.

### 카페

- `GET /cafes/nearby`
- `GET /cafes/{cafeId}`
- `GET /cafes/by-qr/{qrCode}`
- `GET /cafes/{cafeId}/games`

### 게임

- `GET /games`
- `GET /games/{gameId}`
- `GET /games/search`
- `GET /games/{gameId}/similar`

### 이미지 인식

- `POST /recognitions`
- `GET /recognitions/{recognitionId}`
- `POST /recognitions/{recognitionId}/confirm`

### 추천

- `POST /recommendations`
- `GET /users/{userId}/recommendation-profile`

### 사용자 기록

- `POST /events`
- `GET /users/{userId}/history`
- `POST /users/{userId}/played-games`
- `POST /users/{userId}/hidden-games`

### 관리자

- `POST /admin/games`
- `PATCH /admin/games/{gameId}`
- `POST /admin/cafes`
- `PATCH /admin/cafes/{cafeId}`
- `PUT /admin/cafes/{cafeId}/inventory`

관리자 API는 초기에는 간단해도 좋지만, 나중에 카페 직원이나 운영자가 게임 목록과 선반 위치를 관리할 수 있어야 합니다.

## 데이터베이스 설계 요청

아래 테이블 또는 컬렉션을 기준으로 데이터 모델을 설계해주세요.

- `games`
- `game_aliases`
- `game_relations`
- `cafes`
- `cafe_inventory`
- `users`
- `anonymous_sessions`
- `user_events`
- `played_games`
- `hidden_games`
- `recognition_jobs`
- `recognition_candidates`
- `recommendation_logs`
- `admin_users`

각 모델의 필드, 타입, 인덱스, 관계를 제안해주세요.

특히 다음 조회가 빠르게 동작해야 합니다.

- 특정 카페의 보유 게임 목록 조회
- 특정 조건으로 카페 보유 게임 필터링
- 게임명/별칭 검색
- 특정 사용자의 최근 검색/플레이 기록 조회
- 현재 카페 기준 추천 후보 조회
- 이미지 인식 결과 후보 저장 및 확인

## 이미지 저장과 개인정보 원칙

보드게임 사진은 개인정보가 포함될 수 있으므로 보관 정책을 명확히 해야 합니다.

기본 원칙은 다음과 같습니다.

- 사진은 인식 처리 후 기본적으로 장기 보관하지 않습니다.
- 품질 개선을 위해 저장할 경우 사용자 동의를 받아야 합니다.
- 사람 얼굴이나 민감 정보가 포함될 가능성을 고려합니다.
- 이미지 원본, 썸네일, AI 입력용 전처리 이미지의 보관 기간을 분리해 설계합니다.
- 인식 로그에는 원본 이미지가 아니라 결과와 후보 중심의 데이터를 남깁니다.

## 인증과 사용자 식별

초기 MVP에서는 로그인 없이 사용할 수 있어야 합니다.

다만 추천 기록을 쌓기 위해 다음 구조를 고려해주세요.

- 익명 세션 ID 기반 사용
- 선택적 소셜 로그인
- 기기 변경 시 기록 이전 가능성
- 사용자 기록 삭제 기능
- 위치 권한 거부 시 수동 카페 선택

운영자 또는 관리자 기능은 일반 사용자와 분리된 인증이 필요합니다.

## 기술 스택 제안 요청

아래 조건을 고려해 백엔드 기술 스택을 추천해주세요.

- 빠르게 MVP를 만들 수 있어야 합니다.
- 모바일 앱과 웹 프로토타입에서 쉽게 호출할 수 있어야 합니다.
- 이미지 업로드와 외부 AI API 연동이 쉬워야 합니다.
- 추천 로직을 점진적으로 개선할 수 있어야 합니다.
- 카페/게임 데이터가 늘어도 유지보수하기 쉬워야 합니다.

가능한 조합 예시는 다음과 같습니다.

- Node.js + NestJS 또는 Express
- Python + FastAPI
- PostgreSQL
- Redis
- S3 호환 이미지 저장소
- 외부 이미지 인식 API 또는 Vision 모델

각 선택지의 장단점과 MVP에 추천하는 조합을 제안해주세요.

## 중요한 설계 원칙

- 앱은 현장에서 빠르게 쓰이는 도구이므로 API 응답은 빠르고 간결해야 합니다.
- 보드게임 설명과 카페 보유 데이터는 서버의 단일 출처가 되어야 합니다.
- 추천은 반드시 현재 카페 보유 게임 여부를 우선 고려해야 합니다.
- 이미지 인식 결과는 신뢰도와 후보 목록을 함께 반환해야 합니다.
- 로그인 없이도 기본 사용이 가능해야 합니다.
- 사용자의 사진과 위치 정보는 최소 수집, 짧은 보관, 명확한 동의를 기본으로 합니다.
- 초기에는 단순한 구조로 시작하되, 카페와 게임 데이터가 늘어날 수 있는 확장성을 고려합니다.
- 관리자 기능은 나중에 붙일 수 있도록 데이터 모델과 권한 구조를 처음부터 분리합니다.
